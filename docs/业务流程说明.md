# 挂号核心流程图及规则

> 更新时间：2026-04-27  
> 代码基准：`controller + service + mapper + WebSecurityConfig + db.sql`

## 1. 适用范围

本文聚焦在线挂号主链路：

- 患者端：登录、选择科室/医生、查询排班、提交预约、取消预约、查询预约
- 医生端：查看预约、完成就诊
- 管理端：排班创建/维护（影响可预约号源）

## 2. 挂号主流程（患者端）

```mermaid
flowchart TD
    A[患者访问前端] --> B{是否已登录}
    B -->|否| C[登录/注册]
    B -->|是| D[浏览科室和医生]
    C --> D
    D --> E[查看医生排班 GET /api/schedule/{doctorId}]
    E --> F[选择排班与就诊人]
    F --> G[提交预约 POST /api/appointment]
    G --> H{后端规则校验}
    H -->|失败| I[返回业务错误 code!=0]
    H -->|通过| J[原子扣减号源 remaining_amount-1]
    J --> K[创建预约记录 t_appointment]
    K --> L{剩余号源是否为0}
    L -->|是| M[排班状态置为 FULL]
    L -->|否| N[保持 AVAILABLE]
    M --> O[返回预约成功]
    N --> O
```

## 3. 取消预约流程（患者端）

```mermaid
flowchart TD
    A[我的预约 GET /api/appointment] --> B[发起取消 POST /api/appointment/{id}/cancel]
    B --> C{预约是否属于当前用户}
    C -->|否| X[NO_AUTH_ERROR]
    C -->|是| D{状态是否 PENDING}
    D -->|否| Y[OPERATION_ERROR]
    D -->|是| E{是否在创建后30分钟内}
    E -->|否| Z[OPERATION_ERROR: cancel window expired]
    E -->|是| F[回补号源 remaining_amount+1]
    F --> G{排班状态为FULL且余号>0}
    G -->|是| H[状态改回 AVAILABLE]
    G -->|否| I[保持原状态]
    H --> J[预约状态改为 CANCELLED 并记录取消原因/时间]
    I --> J
    J --> K[返回取消成功]
```

## 4. 医生完成就诊流程

```mermaid
flowchart TD
    A[医生登录] --> B[POST /api/doctor/appointments/{id}/complete]
    B --> C{当前账号是否绑定医生档案}
    C -->|否| X[NOT_FOUND_ERROR]
    C -->|是| D{预约是否存在且属于该医生}
    D -->|否| Y[NOT_FOUND_ERROR/NO_AUTH_ERROR]
    D -->|是| E{预约状态是否 PENDING}
    E -->|否| Z[PARAMS_ERROR]
    E -->|是| F[更新状态为 COMPLETED]
    F --> G[返回 completed]
```

## 5. 核心业务规则（按当前实现）

### 5.1 准入与权限

- 患者预约相关接口 `/api/appointment/**`、就诊人接口 `/api/patient/**` 需要 `ROLE_PATIENT`。
- 医生工作台 `/api/doctor/profile`、`/api/doctor/statistics`、`/api/doctor/appointments/**` 需要 `ROLE_DOCTOR`。
- 排班管理 `/api/admin/schedule/**` 需要 `ROLE_ADMIN`。
- 公开查询接口：科室、医生、排班、公告、今日统计（GET）。

### 5.2 预约创建规则

提交 `POST /api/appointment` 时，后端按顺序校验：

1. 参数完整：`doctorId/scheduleId/patientId` 必填。
2. 当前登录用户存在。
3. 就诊人存在，且归属当前用户。
4. 医生存在，且 `departmentId` 不为空。
5. 排班存在，且属于该医生。
6. 排班日期不能是过去日期。
7. 排班不能是 `SUSPENDED`，且余号必须大于 0、状态不能是 `FULL`。
8. 同就诊人同科室同日不可重复预约（应用层计数检查 + 数据库唯一索引兜底）。

通过后执行：

- SQL 原子扣减：`remaining_amount > 0` 条件更新。
- 新增预约记录：状态 `PENDING`，通知标记 `notification_sent=false`。
- 如果余号扣减后为 0，自动将排班状态更新为 `FULL`。

### 5.3 预约取消规则

- 仅允许取消自己的预约。
- 仅 `PENDING` 状态可取消。
- 仅允许在预约创建后 30 分钟内取消。
- 取消成功后自动回补号源；若排班原为 `FULL` 且回补后余号>0，则状态切回 `AVAILABLE`。
- 记录 `refuse_time` 和 `refuse_reason`。

### 5.4 排班管理规则（管理员）

- 创建排班：医生必须存在；同一医生同日同时间段不允许重复（应用层查重 + `uk_doctor_date_slot`）。
- 更新排班：`totalNum >= 已使用号源`，否则拒绝。
- 删除排班：若已使用号源 > 0（已有预约占用），则拒绝删除。
- 手工增减余号接口会触发 `FULL/AVAILABLE` 状态联动。

### 5.5 就诊人规则

- 就诊人必须通过姓名、身份证、手机号、性别校验。
- 身份证唯一性是“同用户维度”校验（`user_id + id_card` 逻辑）。
- 第一个就诊人默认设为 `is_default=true`；若新建/编辑时指定默认，会先清空该用户其他默认项。
- 删除就诊人时代码未做“是否存在预约”前置业务校验，主要依赖数据库外键约束。

## 6. 数据一致性与并发策略

### 6.1 已落地的一致性手段

- 事务：创建预约、取消预约、排班增减/更新/删除都在事务中执行。
- 原子库存扣减：`UPDATE t_schedule ... WHERE remaining_amount > 0`。
- 状态联动：号源与 `AVAILABLE/FULL` 双向同步。
- 约束兜底：
  - `uk_doctor_date_slot` 防重复排班
  - `uk_patient_dept_date` 防同日同科室重复预约
  - 外键保证主从记录完整性

### 6.2 当前实现边界

- `queueNumber` 通过“当前有效预约数 + 1”计算，极高并发下可能出现并列号风险。
- `uk_patient_dept_date` 不区分状态，取消后同日同科室默认不可再次预约。
- `t_appointment.status` 在数据库层含 `NO_SHOW`，但当前业务流主要使用 `PENDING/COMPLETED/CANCELLED`。

## 7. 状态机（当前实现）

### 7.1 排班状态机

```text
AVAILABLE --(余号扣减到0)--> FULL
FULL --(取消预约回补余号>0)--> AVAILABLE
AVAILABLE/FULL --(管理员设置)--> SUSPENDED
SUSPENDED --(管理员设置)--> AVAILABLE 或 FULL
```

### 7.2 预约状态机

```text
PENDING --(医生完成就诊)--> COMPLETED
PENDING --(患者30分钟内取消)--> CANCELLED
COMPLETED/CANCELLED --(终态)--> 不再流转
```

## 8. 典型失败场景

| 场景 | 返回码/信息（示例） |
| --- | --- |
| 排班无余号 | `50001` / `no remaining quota` |
| 排班停诊 | `50001` / `schedule is suspended` |
| 重复预约（同就诊人同科室同日） | `50001` / `the patient already has an appointment...` |
| 超过取消窗口 | `50001` / `cancel window expired` |
| 越权操作他人预约 | `40101` / `no permission` |
| 医生完成非本人预约 | `40101` / `no permission` |
